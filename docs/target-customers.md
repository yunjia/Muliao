## 画像 C — 丽姐：独立经营者（智能销售助手）

> **MVP 首选画像** — 痛点尖锐、ROI 可量化、天然适合多 Agent 架构。

### 基本信息

- **年龄**：35 岁
- **职业**：独立经营者，销售自有产品/服务，无专职销售团队
- **技术水平**：非技术背景，日常使用 WhatsApp / 微信 / Telegram 与客户沟通
- **工作节奏**：同时跟进 20-50 个潜在客户，沟通分散在多个渠道

### 核心痛点

1. **售前沟通极其耗时** — 成交一单之前要经历多轮、长周期的对话：解答产品疑问、发送资料、跟进意向。这些沟通占据了她大部分工作时间
2. **沟通量 ≠ 成单率** — 很多精力花在最终不会成交的客户身上，但在早期很难分辨谁是"真买家"。聊得多不代表成单概率高
3. **客户意向分析靠直觉** — 判断客户是否真的有购买意向、购买能力，需要从对话潜台词中分析（比如反复问价格细节可能是高意向，只问基本信息可能只是随便看看）。人工做这件事既主观又低效

### 期望的 AI 助手能力

- **代为初步沟通**：AI 助手能自动回复潜客的产品咨询、解答常见问题、发送产品资料、进行温和推销
- **客户画像与分级**：从对话内容中分析客户的购买意愿、购买能力、紧迫度，自动给客户打标签（🔴高意向 / 🟡观望 / 🟢低优先级）
- **智能精力分配**：对低价值客户自动降低跟进频率，对高价值客户加强跟进并及时提醒丽姐
- **关键时刻人工接管**：当客户进入 close deal 阶段（讨价还价、确认合同细节），立即通知丽姐亲自介入完成成交

### 一日场景：丽姐的周三

早上 8 点，丽姐打开 WhatsApp，看到 Hermes 的晨间汇报：

> ☀️ 昨晚 + 今早共有 **5 位客户** 发来消息，我已代为回复。重点如下：
>
> 🔴 **张总**（高意向）— 追问了产品定价细节和交付周期，还问了大批量采购是否有折扣。建议你**今天亲自跟进** → [查看对话摘要]
>
> 🟡 **李女士**（观望）— 询问了基本功能，未深入。我已发送产品手册，会持续保持联系
>
> 🟡 **王先生**（观望）— 上周聊过一轮后沉默三天，昨晚重新回复问了售后政策。意向有回升迹象
>
> 🟢 **赵总 / 刘经理**（低优先级）— 只问了"价格多少"就没下文，可能在比价。已排入低频跟进

丽姐点开张总的对话摘要，看到一张客户画像卡片：

| 维度 | 分析 |
|------|------|
| 购买意愿 | ⬆️ 高 — 主动追问定价、交付、折扣 |
| 购买能力 | 中高 — 提到公司规模 50+ 人，问批量价 |
| 紧迫度 | 中 — 问了"最快什么时候能到货"但未催 |
| 决策链 | 可能需内部审批 — 说了"我回去跟领导商量" |

丽姐直接拿起手机给张总发了条语音，从折扣方案切入，半小时后拿到了口头意向。

下午，Hermes 又推送了一条消息："王先生刚才又问了参考案例，意向等级从 🟡 上调至 🔴，建议你今明两天找时间聊一下。"

到晚上，丽姐算了一下：今天自己只亲自沟通了 2 个客户，但都是高概率成交的。剩下十几个客户的日常跟进，全由 AI 完成。

### 关键决策因素

| 会选择 Muliao 的原因 | 可能放弃的原因 |
|----------------------|--------------|
| 能自动筛选高价值客户，不再盲目跟进 | 担心 AI 代为沟通会"说错话"丢客户 |
| 节省 70-80% 售前沟通时间 | 不确定 AI 是否真的能判断购买意向 |
| 关键时刻可以一键接管，不脱离实际交易 | 客户发现是 AI 在聊可能会反感 |
| 客户画像卡片让决策有据可依 | 初期配置产品知识库可能有门槛 |

### Muliao 能力映射

| 丽姐的需求 | Muliao 能力（参考 [design.md](design.md)） |
|-----------|-------|
| 同时跟进多个客户的独立对话 | **多 Agent 路由**：每个客户路由到独立会话，上下文隔离 |
| AI 自动回复产品咨询 | **自定义 Skill**：产品知识库 Skill + 销售话术模板 |
| 从对话分析客户画像 | **子代理**：分析 Agent 读取对话记录，输出结构化画像 |
| 客户分级与跟进频率调整 | **心跳任务**：定期扫描客户状态，对低优先级客户降频跟进 |
| 高意向时通知人工接管 | **多渠道消息推送**：触发条件满足时推送提醒到丽姐的私人频道 |
| 丽姐随时接管对话 | **权限控制**：人工接管后 AI 退出该客户会话，丽姐直接对话 |

## 画像 B — 文哲：研究生（学习 / 科研资料助手）

> **MVP 首选画像** — 低风险、已有[详细场景设计](scenarios/research-assistant/个人研究助手.prompt.md)可直接复用。

### 基本信息

- **年龄**：25 岁
- **职业**：计算机方向硕士在读，研究方向为机器人 / 具身智能
- **技术水平**：有一定技术基础（写代码、用 Git），但不想花时间折腾工具链配置
- **工作节奏**：每天需要跟踪 2-3 个细分领域的最新论文，阅读 + 整理笔记

### 核心痛点

1. **论文追踪费时** — 每天手动刷 arXiv、Google Scholar、Twitter，生怕漏掉重要论文，但大部分时间花在了筛选上而非阅读上
2. **资料跨平台分散** — 论文 PDF 在 Zotero，笔记在 Notion，代码 bookmark 在浏览器，讨论在微信群。找东西时到处翻
3. **阅读笔记整理低效** — 读完一篇论文后，想记录关键点和与自己研究的关联，但手动整理摘要和标签太花时间

### 期望的 AI 助手能力

- **每日自动推送**：根据自己的研究兴趣（如 "diffusion policy""robot manipulation"），每天推送 arXiv 新论文的中文摘要
- **自动去重**：已经看过 / 推送过的论文不要重复出现
- **兴趣自适应**：读了什么、标记了什么，助手能逐步理解自己的偏好，调整推送范围
- **快速提问**：发一条消息就能问"这周 CVPR 方向有什么新的 diffusion 论文？"，不用自己构造搜索 query

### 一日场景：文哲的周一

早上 9 点，文哲打开 Telegram，看到 Hermes 昨晚推送的消息：

> 📚 **arXiv 日报 — 03/20（周五）**
>
> 你关注的领域今天有 **8 篇新论文**，其中 3 篇与你的方向高度相关：
>
> 1. **DiffGrasp: Diffusion-based 6-DoF Grasp Pose Generation**
>    *Keywords: diffusion policy, grasping, 6-DoF*
>    要点：提出了基于扩散模型的抓取姿态生成方法，在 GraspNet 上 SOTA…
>    → [arXiv 链接] [PDF]
>
> 2. **EmbodiedGPT-2: Scaling Vision-Language-Action Models**
>    *Keywords: embodied AI, VLA, scaling*
>    要点：在 EmbodiedGPT 基础上扩展到 13B 参数…
>    → [arXiv 链接] [PDF]
>
> *（另外 5 篇相关度较低，点此展开…）*

文哲扫了一眼，觉得第一篇和自己的毕设直接相关，回了一条消息："帮我详细总结一下 DiffGrasp 这篇的方法和实验部分"。两分钟后收到 Hermes 的结构化摘要。

下午组会前，文哲给 Hermes 发了一条："帮我整理一下这周看的论文，做个简单的进展汇报"。Hermes 根据这周的推送记录和文哲的阅读标记，生成了一份 Markdown 格式的周报草稿。

### 关键决策因素

| 会选择 Muliao 的原因 | 可能放弃的原因 |
|----------------------|--------------|
| 每天自动推送，不用自己刷 arXiv | 摘要质量不够好，还是得自己读原文 |
| 中文摘要降低阅读门槛 | 推送太多造成信息过载 |
| 在 Telegram 里就能完成，不用装新工具 | 自己已有成熟的论文管理工作流 |
| 能记住自己的兴趣，推送越来越精准 | 对 arXiv 之外的来源（会议、预印本）覆盖不足 |

### Muliao 能力映射

| 文哲的需求 | Muliao 能力（参考 [design.md](design.md)） |
|-----------|-------|
| 每日自动推送论文摘要 | **Cron 定时任务** + **自定义 Skill**（arXiv 搜索 + 摘要生成） |
| 去重与已读记录 | **工作区文件管理**：维护 `papers-read.md` 去重文件 |
| 兴趣自适应 | **透明记忆**：`core-interests.md` 可由用户直接编辑，Agent 也可自动更新 |
| 在 Telegram 里交互 | **多渠道网关**：Telegram Channel 推送 + 私聊对话 |
| 论文详细摘要 | **子代理**：摘要 Agent 调用 Browser 工具读取 PDF 并生成结构化总结 |

## 画像 A — 小雅：职场白领（个人效率助手）

> **模板沉淀阶段（Phase 3）** — 场景通用、受众广，适合成为成熟模板而非 MVP 突破口。

### 基本信息

- **年龄**：28 岁
- **职业**：互联网公司运营经理，管理跨部门项目
- **技术水平**：非技术背景，日常用飞书 / 企微 / 邮件 / Excel
- **工作节奏**：每天处理 50+ 封邮件、3-5 个会议、多个并行项目的待办

### 核心痛点

1. **邮件堆积** — 每天大量邮件中混杂着重要审批、普通通知和垃圾邮件，分拣本身就是一项工作
2. **日程碎片化** — 会议被插来插去，经常漏掉待办或重复安排时间
3. **待办散落在多个工具** — 飞书里有一些、邮件里有一些、微信群里被 @ 了几件事、自己的笔记本上还有一些，汇总就要花半小时

### 期望的 AI 助手能力

- **晨间摘要**：每天早上推送当日日程 + 重点邮件 + 待办汇总
- **邮件分拣**：自动分类（需回复 / 仅通知 / 可忽略），重要邮件提醒
- **待办聚合**：从邮件、日历、聊天中提取待办事项，统一管理
- **会议准备**：会议前 15 分钟推送相关材料和上次会议纪要的要点

### 一日场景：小雅的周二

早上 8:30，小雅在飞书上收到 Hermes 的消息：

> ☀️ **今日概况**
>
> 📅 今天有 **4 个会议**，第一个是 10:00 的产品评审
> 📧 昨晚收到 **23 封邮件**，其中 3 封需要你回复：
> - 市场部 Alice 问本月活动排期（建议今天回复）
> - 财务找你确认 Q2 预算表（deadline 周四）
> - VP 转发了竞品分析报告（FYI，已归档）
> ✅ 待办汇总：**7 项**，其中 2 项今天到期

小雅回了一句"帮我给 Alice 回复下，就说排期周四前给她"。Hermes 草拟了一封简洁的回复，小雅看了下点了确认发送。

下午 2 点，会议前 Hermes 推送："2:30 的产品评审，这是上次评审的 3 个 Action Item 进展和今天的议程。" 小雅带着准备好的材料走进会议室。

### 关键决策因素

| 会选择 Muliao 的原因 | 可能放弃的原因 |
|----------------------|--------------|
| 晨间摘要节省每天 30 分钟 | 公司邮箱/日历权限申请可能很麻烦 |
| 在飞书里就能用，不用开新应用 | 担心 AI 看到敏感邮件内容 |
| 帮代拟邮件但需确认才发送，安全感足 | 已有的效率工具（如 Notion AI）可能够用了 |
| 待办聚合能减少"到处翻"的焦虑 | IT 部门可能不允许接入第三方工具 |

### Muliao 能力映射

| 小雅的需求 | Muliao 能力（参考 [design.md](design.md)） |
|-----------|-------|
| 晨间摘要推送 | **心跳任务**：每天早上触发，汇总邮件/日历/待办 |
| 邮件分拣与回复 | **自定义 Skill**：邮件分类 Skill + 草拟回复 Skill |
| 待办聚合 | **子代理**：从多个来源提取任务项，写入统一的待办文件 |
| 会议准备推送 | **定时任务**：会议前 N 分钟触发，读取上次纪要 |
| 在飞书中交互 | **多渠道网关**：Feishu 渠道接入 |

## 画像 D — 老陈：技术探索者（OpenClaw 产品化）

> **生态社区阶段（Phase 5）** — 开发者群体，为 Muliao 贡献模板和最佳实践。
>
> ⚠️ **此画像为初稿，随 Muliao 项目推进很可能大改。** 当前基于已有信息勾勒，待 MVP 阶段（丽姐/文哲场景）落地后，再根据实际技术栈选型、社区反馈和开发者参与情况重新评估。

### 基本信息

- **年龄**：32 岁
- **职业**：全栈开发者，对多 Agent 系统和 AI 应用感兴趣
- **技术水平**：高——熟悉 Docker、CLI、API 开发、配置文件编写
- **动机**：想基于 OpenClaw 为自己的客户搭建 AI 助手方案，或做独立产品

### 核心痛点

1. **OpenClaw 学习曲线陡** — 文档虽全但内容分散，从零搭建一个可用的多 Agent 系统需要大量试错
2. **缺少可参考的产品化案例** — 社区有很多技术 demo，但很少有面向真实用户的完整配置模板
3. **配置组合爆炸** — Agents、Skills、路由、权限、定时任务的组合方式太多，不知道什么是最佳实践

### 期望的 Muliao 价值

- **开箱即用的配置模板**：fork Muliao 仓库，改几个配置文件就能跑起来
- **清晰的架构参考**：SOUL.md 怎么写、多 Agent 怎么分工、路由怎么配——有实际案例可参照
- **可二次定制的脚手架**：不是一个封闭产品，而是可以在上面搭建自己方案的基础

### 一日场景：老陈的周末

周六下午，老陈想给自己的一个客户做一个"客服 AI 助手"原型。他 fork 了 Muliao 仓库，看了一下 `teams/default/workspace/` 的结构。

`SOUL.md` 给了他清晰的角色定义参考——原来人格设定要这样写。`AGENTS.md` 展示了主 Agent 和子代理的配置关系。`skills/` 目录下的示例 Skill 让他明白了自定义能力的写法。

他把 `SOUL.md` 改成了客服人格，把产品知识库写成了一个 Skill，30 分钟后用 `docker/run.sh` 跑起来了一个能在 Telegram 上回复客户咨询的原型。周日上午他加了一个分析 Agent 做对话满意度评分，提交了 PR 回 Muliao 社区。

### 关键决策因素

| 会选择 Muliao 的原因 | 可能放弃的原因 |
|----------------------|--------------|
| 模板和文档质量高，减少从零摸索的时间 | 如果 Muliao 和上游 OpenClaw 更新不同步就很头疼 |
| 项目结构清晰，易于理解和二次开发 | 如果社区不活跃，模板可能会过时 |
| Docker 一键运行，本地调试方便 | 如果过于固定某种架构，灵活性可能不够 |

### Muliao 能力映射

| 老陈的需求 | Muliao 提供的价值 |
|-----------|-------|
| 快速搭建多 Agent 原型 | **Docker 脚本** + **团队配置模板**（`teams/` 目录结构） |
| 角色设计参考 | **SOUL.md / IDENTITY.md 示例** + 设计理念文档 |
| Skill 开发指南 | **skills/ 目录** 中的示例 Skill + SKILL.md 规范 |
| 社区分享与协作 | **开源模板** + 贡献指南（Phase 5 规划） |

## MVP 优先级矩阵

### 为什么 MVP 选丽姐 + 文哲？

| 维度 | 丽姐（智能销售助手） | 文哲（科研资料助手） |
|------|---------------------|---------------------|
| **痛点尖锐度** | 🔴 极高 — 直接影响收入 | 🟡 中高 — 影响效率但非生死 |
| **ROI 可量化** | ✅ 节省沟通时间 × 成单率提升 | ✅ 节省每天 30-60 分钟筛选时间 |
| **Muliao 架构匹配度** | ✅ 天然多 Agent — 沟通 / 分析 / 提醒 Agent 各司其职 | ✅ 单 Agent + Skill 即可 MVP |
| **已有设计资产** | ❌ 需新建场景模板 | ✅ 已有[详细设计](scenarios/research-assistant/个人研究助手.prompt.md) |
| **核心定位契合** | ✅ 完美体现"AI 不只是聊天，而是执行者" | ✅ 体现"主动性"和"记忆成长" |

### MVP → 模板 → 生态 的路径

```
Phase 1-2 (MVP)
  ├── 丽姐场景 → 智能销售助手模板 → 验证多 Agent 路由 + 客户画像 + 人工接管
  └── 文哲场景 → 科研资料助手模板 → 验证定时任务 + Skill + 记忆成长

Phase 3 (模板沉淀)
  └── 小雅场景 → 个人效率助手模板 → 复用 MVP 验证的基础设施

Phase 5 (生态社区)
  └── 老陈 → 开发者贡献模板、最佳实践、二次定制方案
```

## 需要注意的产品设计问题

### 丽姐场景特有问题

1. **AI 身份披露**：AI 代表丽姐与客户沟通时，是否需要告知客户对方是 AI？这涉及 Muliao"强安全感"原则的外延——不只是对用户透明，也要考虑对用户的客户的透明度。建议 MVP 阶段默认以丽姐的名义沟通，但在 SOUL.md 中提供配置选项
2. **"说错话"的风险控制**：需要设计"敏感操作确认"机制——比如涉及报价、承诺交付时间等关键话术时，AI 暂停并请求丽姐确认后再发送
3. **产品知识库冷启动**：丽姐非技术背景，需要提供聊天式的知识库导入向导（"把你的产品介绍发给我，我来学习"），而非让她编辑 Skill 文件

### 跨画像的共性问题

4. **渠道接入门槛**：所有画像都依赖 IM 渠道接入，需确保 Telegram / WhatsApp / 飞书的配置过程对非技术用户足够简单
5. **信任建立周期**：非技术用户（丽姐、小雅）对"AI 动我的东西"天然有戒心，MVP 阶段建议采用"只读 + 建议"模式，逐步升级到"代为执行"
