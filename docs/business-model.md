# Muliao 商业模式：三层架构

> 本文档描述 Muliao 的商业模式设计。技术设计见 [overview.md](overview.md)；目标客户画像见 [target-customers.md](target-customers.md)。

***

## 一、总体架构

Muliao 采用**三层互补**的商业架构。三层不互斥，分阶段叠加：

```
┌─────────────────────────────────────────────────────────┐
│  第三层：母公司纵深应用                                    │  ← 核心收入
│  基于 Muliao 做垂直场景 SaaS 产品                         │
│  第一个产品：智能销售助手（丽姐场景）                        │
├─────────────────────────────────────────────────────────┤
│  第二层：Muliao 自养机制（Capture Mechanism）              │  ← 项目可持续
│  Cloud Hosting · Enterprise Support · Skill Marketplace │
│  认证培训 · 硬件预装                                      │
├─────────────────────────────────────────────────────────┤
│  第一层：Muliao 开源中间件                                │  ← 生态基础
│ ┌───────────────────────────────────────────────────┐   │
│ │  用户体验层（UX Layer）                             │   │
│ │  对话式角色定义 · 自然成长机制                        │   │
│ ├───────────────────────────────────────────────────┤   │
│ │  框架层（Framework Layer）                          │   │
│ │  多 Agent 编排 · 场景模板 · Skills · Docker 部署     │   │
│ └───────────────────────────────────────────────────┘   │
├─────────────────────────────────────────────────────────┤
│  OpenClaw（底层引擎）                                     │
│  Agent 运行时 · 渠道网关 · 工具执行                        │
└─────────────────────────────────────────────────────────┘
```

**核心逻辑**：Muliao 作为开源中间件构建生态基础，自养机制保障项目可持续发展，母公司作为第一个也是最重要的下游用户，从丽姐场景切入产生核心收入，并将实践经验反哺开源。

***

## 核心竞争力

> Muliao 的护城河不是单一技术，而是**面向非技术用户的产品化能力**与**开源生态飞轮**的组合。

### 1. 唯一聚焦「普通人友好」的 AI 助手中间件

市面上的多 Agent 框架（OpenClaw、AutoGen、CrewAI 等）均面向开发者。Muliao 是目前唯一在框架层之上系统性构建**非技术用户体验层（UX Layer）**的开源项目：

- **对话式角色定义**——用户用自然语言描述需求，Muliao 自动生成 SOUL.md / IDENTITY.md / AGENTS.md 等底层配置，开发者仍可直接编辑
- **自然成长机制**——角色通过「使用」而非「配置」进化：纠正即学习、反馈即调整、新能力主动询问是否启用

这意味着：**下游公司基于 Muliao 构建产品时，天然获得面向非技术终端用户的交互能力，无需自己从零搭建。**

### 2. 「幕僚团队」架构——团队长 + 专职幕僚

「幕僚」的本意是一群各有专长的谋士。Muliao 的产品架构忠于这个隐喻：用户拥有一支**可见的、各有名字和性格的 AI 幕僚团队**，而非一个声称什么都能做的全能助手。

- **团队长**（默认入口）：用户发消息不指定谁，就由团队长接。他负责理解需求、分派给合适的幕僚、汇总结果
- **专职幕僚**（小美、小研…）：各有名字、性格和专长——用户也可以直接找他们对话
- 关键区别：**幕僚不是隐形的**——用户知道他们的存在，能看到是谁在做事

这既保留了「不知道找谁就找团队长」的低门槛，又提供了「直接找专人」的高效路径和专业信任感。底层是一套包含**场景模板、角色编排、记忆隔离与共享、团队协作输出**的完整产品化方案。

竞品对比：
| | OpenClaw 原生 | 通用 Agent 框架 | Muliao |
|---|---|---|---|
| 多 Agent 协作 | ✅ | ✅ | ✅ |
| 非技术用户可用 | ❌ 需手写配置 | ❌ 需编程 | ✅ 对话式配置 |
| 场景模板 | ❌ | 部分 | ✅ 一键启用 |
| 可见的专职团队 | ❌ 需自行设计 | ❌ | ✅ 团队长 + 专职幕僚 |
| 渐进式成长 | ❌ | ❌ | ✅ 自然成长机制 |

### 3. 开源中间件定位——生态杠杆而非单点产品

Muliao 不直接与 OpenClaw 竞争，而是站在它之上做**增值层**。这个定位带来三重杠杆：

- **向下兼容**：OpenClaw 的所有能力（渠道网关、工具执行、Sub-Agent）Muliao 自动继承
- **向上赋能**：下游公司无需重复造「对普通人友好」的轮子，直接复用 Muliao 的 UX Layer + 模板体系
- **社区飞轮**：每个下游公司的实践（模板、Skills、最佳实践）都沉淀回开源生态，反哺所有参与者

### 4. 母公司即第一用户——dogfooding 驱动的产品迭代

母公司基于 Muliao 做智能销售助手（丽姐场景），既是核心收入来源，也是最严苛的内部用户。这种 dogfooding 模式确保：

- 框架能力经过**真实商业场景**验证，而非停留在 demo 阶段
- 产品迭代由**真金白银的 ROI 反馈**驱动，而非技术偏好
- 从丽姐场景中沉淀出的模板和 Skills，成为开源生态的**高质量种子内容**

### 5. 数据主权与信任壁垒

Muliao 坚持自托管优先 + 数据完全可导出：

- 用户数据不经过任何第三方平台，记忆文件（memory/）以可读文档形式存储在用户自己的机器上
- 即使使用 Muliao Cloud 托管服务，数据也可随时迁移到自托管，**零锁定**
- 对丽姐而言，数据主权不是她的购买决策因素（她已经在用 WhatsApp / 微信等云服务），但**「AI 不会乱动我的东西」的可控感**是她愿意持续使用的前提——操作日志、二次确认、随时接管，这些才是她真正在意的信任机制
- 数据主权真正构成**准入门槛**的场景是企业客户和下游公司——它们有合规要求（数据不出境、审计追溯），自托管能力是选型时的硬性条件

> **总结**：Muliao 的核心竞争力 = **UX Layer（面向普通人的交互层）** × **开源中间件生态** × **母公司 dogfooding 驱动的真实场景验证**。三者互相强化，形成竞争对手难以快速复制的组合壁垒。

## 二、第一层：Muliao 开源中间件

### 定位

AI 助手团队的**框架层**——站在 OpenClaw 之上，为下游公司和开发者提供构建「面向普通人的 AI 助手产品」所需的组件和服务。

Muliao 与直接使用 OpenClaw 的核心区别在于：不只提供技术框架，还提供**面向非技术用户的交互层**。

### 开源范围

| 组件 | 说明 |
|------|------|
| **框架层** | 多 Agent 编排核心、渠道路由、工作区管理 |
| **场景模板库** | teams/ 结构、SOUL.md / AGENTS.md / IDENTITY.md 模板 |
| **Skills 体系** | 开发规范、示例 Skills、与 ClawHub 的集成 |
| **部署工具链** | Docker 脚本（docker/）、RPi 部署方案 |
| **文档体系** | 设计理念、用户画像、最佳实践 |
| **用户体验层** | 对话式角色定义、自然成长机制（见下文） |

**不开源 / 独立仓库**：母公司的纵深应用产品代码、特定行业的 premium Skills（如高级客户分析模型）。

### 用户体验层（UX Layer）

> 方向暂定，随丽姐场景的实践经验持续迭代。

当前 OpenClaw 的角色配置体系（SOUL.md、IDENTITY.md、AGENTS.md、openclaw.json 等）是为开发者设计的。对丽姐这类非技术用户而言，手写 Markdown、编辑 JSON、理解 Heartbeat 和 Sub-Agent 概念几乎不可能。

Muliao 的 UX Layer 要解决这个问题，包含两个核心能力：

**能力 A：对话式角色定义**

用自然对话「塑造」角色，Muliao 自动生成底层配置文件：

```
用户：我想要一个帮我接待客户的助手
Muliao：好的！先问几个问题——
       1. 你希望它叫什么名字？
       2. 它跟客户说话应该是什么风格？（礼貌正式 / 亲切随和 / 专业简洁）
       3. 你卖的是什么产品？简单描述一下

用户：叫小美吧。亲切随和。我卖手工护肤品...

Muliao：✅ 小美已创建！你可以随时说「让小美更正式一点」来调整。
```

底层自动生成 SOUL.md、IDENTITY.md、AGENTS.md 等配置，开发者仍可直接编辑这些文件。

设计原则：
- **渐进式**：初始只问 3–5 个核心问题
- **随时可调**：通过对话微调（「别主动推荐高价产品」）
- **透明可查**：用户可以问「小美现在的性格设定是什么？」
- **底层兼容**：生成的文件格式与手写完全一致

**能力 B：自然成长机制**

角色通过「使用」而非「配置」来成长：

- **从对话中学习**——用户纠正 AI 的错误回答，自动记入记忆，无需编辑文件
- **从反馈中调整**——用户说「你跟张总说话太随意了」，AI 更新沟通策略
- **从模板中进化**——Muliao 发现可安装的新能力，用人话询问用户是否启用

### 对下游公司的价值

- 大幅降低构建 AI 助手产品**从 0 到 1** 的成本
- 获得社区贡献的场景模板和 Skills
- 站在成熟的多 Agent 架构上做产品创新
- 直接复用 UX Layer，让自己的产品也能面向非技术用户

**类比**：WordPress（开源 CMS）→ 下游做主题 / 插件 / SaaS；Kubernetes → 下游做托管服务。Muliao 扮演类似角色，但聚焦于 AI 助手领域。

## 三、第二层：Muliao 自养机制

### 目标

让开源项目产生足够资源维持核心团队、基础设施和社区运营。

### 机制设计

五种机制**不互相冲突**，按生态成熟度分阶段启动：

| 机制 | 模式 | 启动时机 |
|------|------|---------|
| **Muliao Cloud** | 托管的 Muliao 实例服务（WordPress.com 模式）；用户无需自行部署，按团队 / 用量付费 | Phase 2–3 |
| **Enterprise Support** | 面向企业的技术支持、咨询、定制开发 | Phase 3+ |
| **Skill Marketplace** | 运营 premium Skill 市场，对付费 Skill 收取分成 | Phase 3–4 |
| **认证 & 培训** | Muliao Certified Partner / Developer 认证体系 | Phase 4–5 |
| **硬件预装** | RPi 5 预装 Muliao 的即插即用设备；硬件利润 + 服务订阅（[部署方案](deploy/RPi%205%20部署方案.prompt.md)已有设计） | Phase 2+ |

### 关键原则

- **核心功能不收费**——避免 open-core 陷阱，不把必要功能藏在付费墙后面
- **Cloud 是 convenience, not lock-in**——用户随时可迁移到自托管，数据完全可导出
- **自养收入优先投入核心开发和社区**，而非利润最大化
- **早期无需自养**——MVP 和 PMF 阶段由母公司收入资助，自养机制在生态成形后才有意义

## 四、第三层：母公司纵深应用

### 定位

母公司是 Muliao 的**第一个也是最重要的下游用户**。它基于 Muliao 做垂直场景的 SaaS 产品，验证框架能力、积累行业经验，并以商业收入反哺开源项目。

### 第一个产品：智能销售助手

> 基于丽姐画像设计，详见 [target-customers.md](target-customers.md)。

**目标用户**：独立创业者 / 小微企业主（无专职销售团队，每天需跟进 20–50 个潜在客户）

**核心功能**：

| 功能 | 说明 |
|------|------|
| AI 自动接待 | 多渠道（WhatsApp / Telegram / 微信）自动回复客户咨询、发送资料 |
| 客户画像分析 | 基于对话内容分析购买意向、购买力、紧急程度，自动分级（🔴高意向 / 🟡观望 / 🟢低优先） |
| 智能跟进 | 根据客户分级自动调节跟进频率，低优先客户减少打扰，高意向客户加强跟进 |
| 关键时刻提醒 | 客户进入成交阶段（议价、确认合同）时，通知用户亲自接管 |
| 每日简报 | 每天早上推送客户动态摘要：谁是高意向、谁在升温、谁需要你出面 |

**收入模式**：SaaS 订阅，按客户数 / 会话量分层定价

**竞争优势**：

- **多 Agent 架构**，而非单一 chatbot——接待、分析、通知由不同 Agent 协作完成
- **对话深度理解**——客户画像基于多轮对话语义分析，非简单关键词匹配
- **透明可控**——用户随时接管，AI 在敏感操作（报价、承诺交期）前暂停等待确认
- **持续学习**——越用越了解用户的产品和客户群

### 为什么从丽姐开始

| 维度 | 理由 |
|------|------|
| 痛点尖锐度 | 直接影响收入——售前沟通占据大量时间，但大部分精力花在低意向客户上 |
| ROI 可量化 | 节省沟通时间 × 成交率提升 = 可衡量的金钱价值 |
| 架构契合度 | 天然需要多 Agent 协作（接待 Agent + 分析 Agent + 通知 Agent） |
| 付费意愿 | 直接帮她赚更多钱的工具，付费意愿远高于效率工具 |
| 验证深度 | 场景足够复杂，能充分检验 Muliao 的 UX Layer、多 Agent 编排、记忆系统 |

### 后续产品线

按验证顺序扩展：

1. **智能销售助手**（丽姐）→ MVP，核心收入来源
2. **研究资源助手**（文哲）→ 可作为免费 / 低价产品引流，同时验证技术
3. **个人效率助手**（小雅）→ Phase 3，通用场景，广覆盖
4. 更多垂直场景 →  根据市场信号选择

## 五、三层飞轮

```
母公司做丽姐产品 ──→ 验证场景，积累实战经验
       │                         │
       │                         ↓
       │              经验沉淀回 Muliao 开源
       │              （模板、Skills、UX 优化）
       │                         │
       ↓                         ↓
  产生商业收入 ──→        开源吸引开发者 & 下游公司
       │                         │
       ↓                         ↓
  资助 Muliao 核心开发    社区贡献新 Skills、模板、bug fix
       │                         │
       └───── 母公司获得更好的框架 ←┘
              做新产品成本更低
```

- **母公司是 Muliao 最大的 dogfooding 用户**——确保开源项目解决真实问题，而非仅服务于理论场景
- **开源生态反哺母公司**——社区贡献的 Skills 和模板降低母公司新产品开发成本
- **自养机制在飞轮加速后才有意义**——不急于在早期做 Cloud / Marketplace，先用母公司收入养活项目

### 冷启动期：飞轮不是一开始就转得起来

飞轮的逻辑链条是通的，但各环节的强弱差异很大：

**强环节（确定性高）**：
- **母公司收入 → 资助 Muliao 开发**：只要丽姐产品能赚钱，开源项目就有资金，不依赖外部社区的善意。Red Hat、Canonical 都是这个模式
- **母公司做产品 → 倒逼框架改进**：自己用自己的框架，痛点暴露最快，这是 dogfooding 的核心价值

**弱环节（需要验证）**：
- **「经验沉淀回开源」不会自动发生**——母公司天然倾向把最有价值的部分留给自己（那是竞争壁垒）。开源回去的是通用部分，而通用部分对下游的吸引力有多大，取决于 Muliao 作为中间件的独立价值。UX Layer 是关键差异化，但目前还是暂定方向
- **「开源吸引开发者」是最不确定的环节**——开发者会问：为什么用 Muliao 而不是直接用 OpenClaw？必须有明确的、OpenClaw 不提供的能力（成熟的 UX Layer、经过实战验证的场景模板库），才能构成选择理由
- **「社区贡献反哺母公司」需要时间和规模**——从「有人用」到「有人贡献高质量 Skills 和模板」，中间隔着很远的路

**现实路径**：早期不是三层同时飞转，而是单引擎驱动——

```
母公司做丽姐 → 赚钱 → 养活 Muliao 开发 → 开源是长线投资
```

飞轮真正开始转，需要两个前提条件先满足：
1. **丽姐产品验证成功**——有收入，有可沉淀的经验
2. **Muliao 相对 OpenClaw 形成清晰的独立价值**——开发者有理由选 Muliao 而非直接用 OpenClaw

## 六、阶段路线

| 阶段 | 开源中间件 | 自养机制 | 母公司产品 |
|------|-----------|---------|-----------|
| **现在 → MVP** | 完善核心框架；丽姐场景模板；对话式角色定义 v1；文档 | 无（母公司资助） | 丽姐智能销售助手 MVP |
| **MVP → PMF**\* | 发布 v1.0；吸引早期开发者社区；自然成长机制 v1 | RPi 预装设备试销 | 丽姐产品验证 & 开始收费 |
| **PMF → Scale** | 场景模板库扩充；Skills 生态初步成形 | Muliao Cloud 上线；Skill Marketplace | 文哲 / 小雅产品线扩展 |
| **Scale → Ecosystem** | 成为 AI 助手领域的事实标准 | 认证体系 + Enterprise Support | 多垂直场景产品矩阵 |

> \* **MVP** = Minimum Viable Product（最小可行产品）；**PMF** = Product-Market Fit（产品-市场匹配，即产品真正找到愿意持续付费的目标市场）。

> 路线图与 [overview.md](overview.md) 中的五阶段技术路线对齐：  
> Phase 1–2（MVP + 控制面板）→ Phase 3（模板沉淀）→ Phase 4（体验迭代）→ Phase 5（社区生态）
