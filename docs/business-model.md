# Muliao 商业模式

> 相关文档：[design.md](design.md)（技术设计）、[target-customers.md](target-customers.md)（用户画像）

---

## 总体架构：三层叠加

```
┌───────────────────────────────────────────┐
│  第三层：母公司纵深应用          ← 核心收入  │
│  垂直 SaaS（首个：智能销售助手）            │
├───────────────────────────────────────────┤
│  第二层：自养机制              ← 项目可持续  │
│  Cloud · Enterprise · Marketplace · 硬件  │
├───────────────────────────────────────────┤
│  第一层：Muliao 开源中间件       ← 生态基础 │
│  UX Layer + Framework Layer              │
├───────────────────────────────────────────┤
│  OpenClaw（底层引擎）                      │
└───────────────────────────────────────────┘
```

一句话：**母公司做产品赚钱 → 养活开源 → 开源吸引生态 → 生态降低母公司新产品成本**。

## 核心竞争力

**公式：UX Layer × 开源生态 × dogfooding 验证**

1. **UX Layer**——唯一面向非技术用户的 AI 助手中间件
   - 对话式角色定义：自然语言 → 自动生成 SOUL.md / IDENTITY.md / AGENTS.md
   - 自然成长机制：纠正即学习、反馈即调整，用户无需碰配置文件

2. **「幕僚团队」架构与赫耳墨斯（团队长 + HR）**

   **问题**：定义一个 agent 对所有人都很痛苦——不只是非技术用户，开发者和 power user 也一样。原因有二：
   - 用户知道自己需要什么，但没有系统整理过——需求是模糊的、隐性的
   - 碰到 SOUL.md、skills、tools 这些概念就产生选择困难——不是不懂，是组合太多、不知道怎么选最合适的

   **方案**：用一个稳定可靠的主 agent 来帮用户定义其他 agent。这就是**赫耳墨斯（Hermes）**——幕僚团队的团队长，同时兼任 HR。

   赫耳墨斯的双重角色：
   - **作为团队长**（OpenClaw 标准主 agent 职能）：用户发消息不指定谁，就由赫耳墨斯接；他负责理解需求、分派给合适的幕僚、汇总结果
   - **作为 HR**（Muliao 独有）：帮用户「招聘」新幕僚——通过反复对话提取用户的隐性需求，和用户一起确定新 agent 的核心配置（SOUL、技能、工具、边界等）

   HR 流程：
   1. 用户表达模糊需求（「我需要一个帮我跟客户聊天的助手」）
   2. 赫耳墨斯通过多轮对话提取关键信息——不是一次性问完，而是渐进式追问
   3. 赫耳墨斯将对话结果翻译为 agent 配置（SOUL.md / IDENTITY.md / skills 等），用人话向用户确认
   4. 新幕僚上岗后，用户在实际使用中继续和赫耳墨斯反馈调整（「小美跟客户说话太生硬了」→ 赫耳墨斯更新小美的 SOUL）

   关键区别：幕僚不是隐形的——用户知道团队里有谁、各自擅长什么、是谁在做事。

   > 这是 Muliao 最核心的差异化：不是「提供一个更好的 agent 框架」，而是「提供一个能帮你搭建 agent 团队的 agent」。

3. **开源中间件定位**——站在 OpenClaw 之上做增值层，不竞争
   - 向下继承 OpenClaw 全部能力，向上提供 UX Layer + 场景模板

4. **Dogfooding**——母公司是第一用户，真实场景驱动迭代

5. **数据主权**
   - 对丽姐：核心不是「数据归我」，而是**可控感**（操作日志、二次确认、随时接管）
   - 对企业客户：自托管是合规硬性条件

## 第一层：开源中间件

在 OpenClaw 之上提供 **UX Layer + 框架层**。

**开源范围**：多 Agent 编排、场景模板库、Skills 体系、部署工具链（Docker / RPi）、UX Layer

**不开源**：母公司产品代码、premium Skills

### UX Layer（方向暂定）

**核心问题**：定义 agent 对所有人都很难——不只是「非技术用户看不懂配置文件」这么简单。即使是开发者，面对 SOUL.md 要写什么、skills 该选哪些、tools 权限怎么划，也会陷入选择困难。根本原因是：用户的需求是隐性的（知道要什么，但没整理过），而 agent 配置要求的是显性的、结构化的定义。

**解法**：不让用户直接面对配置，而是让赫耳墨斯（团队长 / HR）通过对话帮用户完成这个翻译过程。

**能力 A：赫耳墨斯驱动的对话式角色定义**

赫耳墨斯作为 HR，通过多轮对话帮用户「招聘」新幕僚：

1. **需求提取**——用户说「我需要一个帮我接待客户的助手」，赫耳墨斯不要求用户填表，而是像面试官一样渐进追问：叫什么名字？什么风格？做什么产品？哪些事能自己决定、哪些必须问你？
2. **配置翻译**——赫耳墨斯将对话结果翻译为底层配置（SOUL.md / IDENTITY.md / AGENTS.md / skills），用人话向用户确认：「小美会用亲切的语气跟客户聊，遇到报价会先问你，对吗？」
3. **迭代调整**——新幕僚上岗后，用户在实际使用中发现问题，随时跟赫耳墨斯说（「小美太话多了」「别主动推荐高价产品」），赫耳墨斯更新配置

设计原则：
- 渐进式——初始只问 3–5 个核心问题，不一次性把用户问烦
- 底层透明——生成的文件格式与手写完全一致，开发者随时可直接编辑
- 人话确认——每次配置变更都用用户能理解的语言做二次确认

**能力 B：自然成长机制** —— 角色通过使用而非配置进化

- 纠正错误 → 自动记入记忆
- 反馈风格 → 赫耳墨斯更新对应幕僚的沟通策略
- 发现新能力 → 赫耳墨斯用人话询问用户是否启用

## 第二层：自养机制

早期无需自养（母公司资助）。按生态成熟度分阶段启动：

| 机制 | 启动时机 |
|------|---------|
| **Muliao Cloud**（托管实例，按量付费） | Phase 2–3 |
| **硬件预装**（RPi 5 即插即用，[方案](deploy/RPi%205%20部署方案.prompt.md)） | Phase 2+ |
| **Enterprise Support** | Phase 3+ |
| **Skill Marketplace**（抽成） | Phase 3–4 |
| **认证培训** | Phase 4–5 |

原则：核心功能不收费；Cloud 不锁定（随时迁移自托管）；收入优先投核心开发。

## 第三层：母公司产品

### 首个产品：智能销售助手（丽姐场景）

> 画像详见 [target-customers.md](target-customers.md)

**目标**：独立创业者 / 小微企业主，每天跟进 20–50 个潜在客户

| 功能 | 说明 |
|------|------|
| AI 自动接待 | 多渠道（WhatsApp / Telegram / 微信）自动回复、发资料 |
| 客户分级 | 对话语义分析 → 🔴高意向 / 🟡观望 / 🟢低优先 |
| 智能跟进 | 按分级自动调节频率 |
| 关键提醒 | 成交阶段通知用户接管 |
| 每日简报 | 客户动态摘要 |

**收入**：SaaS 订阅，按客户数 / 会话量分层

**为什么从丽姐切入**：痛点直接影响收入 → ROI 可量化 → 付费意愿高 → 天然多 Agent 场景 → 场景够复杂能验证框架

### 后续产品线

1. 智能销售助手（丽姐）→ MVP，核心收入
2. 研究资源助手（文哲）→ 引流 + 验证技术
3. 个人效率助手（小雅）→ Phase 3，广覆盖

## 飞轮与冷启动

**强环节**：
- 母公司收入 → 资助开源（不依赖社区善意，Red Hat 模式）
- 母公司做产品 → dogfooding 暴露痛点 → 框架改进

**弱环节（需验证）**：
- 经验沉淀回开源不会自动发生——母公司倾向保留最有价值部分
- 开源吸引开发者最不确定——需要明确的 OpenClaw 不提供的能力
- 社区贡献反哺需要时间和规模

**现实路径**：早期单引擎——

```
母公司做丽姐 → 赚钱 → 养活 Muliao 开发 → 开源是长线投资
```

飞轮启动前提：① 丽姐产品验证成功 ② Muliao 相对 OpenClaw 形成清晰独立价值

## 阶段路线

| 阶段 | 开源中间件 | 自养机制 | 母公司产品 |
|------|-----------|---------|-----------|
| **→ MVP** | 核心框架 + 丽姐模板 + 对话式角色定义 v1 | 无（母公司资助） | 销售助手 MVP |
| **→ PMF** | v1.0 发布 + 自然成长 v1 | RPi 试销 | 产品验证 & 收费 |
| **→ Scale** | 模板库扩充 + Skills 生态 | Cloud + Marketplace | 文哲 / 小雅扩展 |
| **→ Ecosystem** | 事实标准 | 认证 + Enterprise | 多垂直场景矩阵 |
