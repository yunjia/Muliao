# Muliao MVP 开发计划

> 基于 [design.md](../design.md)、[business-model.md](../business-model.md)、[target-customers.md](../target-customers.md)、[RPi 5 部署方案](../deploy/RPi%205%20部署方案.prompt.md) 制定。
> 每个 Iteration 约 3 天，含可验收标准。
> **核心约束：所有功能必须在 RPi 5 上可运行。RPi 是 MVP 的最小部署单元。**

---

## 现状评估

**已有的**：
- Docker 运行环境（Dockerfile + build.sh + run.sh + entrypoint.sh），支持 PUID/PGID 映射
- 单 Agent（Hermes）运行在 OpenClaw 上，已配置 DeepSeek R1 模型
- Telegram + Slack 双渠道已接入并可用
- Hermes 基础人格已定义（SOUL.md / IDENTITY.md / AGENTS.md / USER.md）
- 论文搜索 Skill 方案已设计（`paper-search/SKILL.md` 文档完成，未验证落地）
- 自然成长机制已有详细设计文档（`design/natural-growth.md`）
- `send-file` Skill 已存在（Telegram 发文件的 workaround）

**还没有的**：
- **RPi 5 部署**（Docker Compose 化未完成、ARM64 镜像未验证、cloud-init 镜像未构建、无 systemd 自启）
- **muliao.io 云端 PWA**（未开始：无 WebSocket Relay、无设备配对、无状态仪表盘、无用户账号体系）
- **Device Agent**（RPi 端设备代理未开发，无法与 muliao.io 云端通信）
- **场景模板系统**（无 `templates/` 结构化模板，无模板部署流程）
- 多 Agent 架构（当前只有 Hermes 一个 Agent）
- HR 对话式组建流程（核心差异化，完全未实现）
- 团队花名册 / 可见性（无团队概念）
- 自然成长机制（纯文档阶段）
- 研究助手场景（文哲）Skill 未验证
- 销售助手场景（丽姐）完全未开始

## MVP 完成标准

MVP 覆盖**文哲（研究生）+ 丽姐（独立经营者）** 两个画像，验证 HR 流程和多 Agent 架构的通用性。**所有功能必须在 RPi 5 上稳定运行**（Phase A 团队辅助部署）。MVP 结束时：

1. **RPi 5 可部署**：团队成员可在 30 分钟内完成一台 RPi 5 的烧录→配置→上线全流程
2. **muliao.io 可用**：RPi 设备自动连接云端，团队通过 muliao.io 查看设备状态、远程配置
3. 和 Hermes 对话，通过 HR 流程「招聘」新幕僚（至少完成 2 个不同场景）
4. 查看团队花名册，知道团队里有谁、各自做什么
5. 幕僚实际上岗工作（研究助手：日报/周报/详读；销售助手：自动接待/客户分级/简报）
6. 对幕僚行为提出纠正，Hermes 识别反馈并提议配置变更（自然成长 v0）
7. 至少 1 个场景模板可一键部署
8. RPi 5 连续运行 72 小时无异常，断电重启后自动恢复

## Iteration 1：RPi 5 Phase A 基础部署（Day 4–6）

> **目标**：实现 RPi 5 团队辅助部署的完整流程——烧录→首次启动→Hermes 在 RPi 上通过 Telegram 可用。对标 [RPi 5 部署方案](../deploy/RPi%205%20部署方案.prompt.md) Phase A。

### Day 4：Ubuntu cloud-init 镜像 + 烧录脚本

**作为**团队成员，**我希望**有一个自动化的 RPi 镜像烧录流程，**以便**30 分钟内完成一台新设备的系统准备。

**Tasks**：
- [x] 创建 `deploy/rpi/` 目录
- [x] 编写 `deploy/rpi/user-data`（cloud-init YAML）：
  - 安装 Docker CE（`get.docker.com` 官方脚本）
  - 安装 Tailscale、avahi-daemon
  - 预拉取 `ghcr.io/muliaoio/muliao:latest` ARM64 镜像
  - 内嵌 `muliao.service` systemd 服务（enable，不 start——等数据部署后启动）
  - 支持占位符替换：`__HOSTNAME__`、`__TAILSCALE_AUTHKEY__`、`__SSH_PUBKEY__`、`__TIMEZONE__`
- [x] 编写 `deploy/rpi/flash-ssd.sh`：下载 Ubuntu Server 24.04 ARM64 镜像 → 烧录到 NVMe SSD → 注入 cloud-init
  - 自动下载并缓存 Ubuntu 镜像（`deploy/.cache/`，已 gitignore）
  - 支持 `--tailscale-key`、`--hostname`、`--ssh-key`、`--skip-flash` 等选项
- [x] 编写 `deploy/rpi/muliao.service`：systemd unit file，开机自启 Docker Compose
  - `WorkingDirectory=/home/muliao`、`EnvironmentFile=/home/muliao/.env`
  - `ExecStartPre=docker compose pull` 确保镜像最新
- [x] 编写 `deploy/rpi/deploy-team.sh`：将开发机团队数据部署到 RPi
  - 解决「如何将开发机上运行的 OpenClaw 配置传给 RPi」的问题
  - rsync 同步 `teams/<name>/` → RPi `/home/muliao/.openclaw/`
  - 同步 `docker-compose.yml` + `.env`（自动修正 `MULIAO_DATA_DIR` 路径）
  - 支持 `--start`/`--restart` 一键启动服务
  - 支持 `--dry-run` 预览

> **架构决策**：Docker 镜像（OpenClaw 二进制）通过 `docker pull` 自动获取，无需烧录到系统镜像。
> 团队数据（`teams/<name>/`）通过 `deploy-team.sh` + rsync over Tailscale SSH 部署。
> 两者分离，镜像可独立更新，数据可增量同步。

### Day 5：RPi 首次启动 + 远程管理

**作为**团队成员，**我希望**烧录好的 SSD 插入 RPi 5 后首次开机能自动完成所有初始化，**以便**无需手动 SSH 逐步安装。

**Tasks**：
- [x] 实际硬件测试：烧录 SSD → 插入 RPi 5 → 接网线通电 → cloud-init 执行完成
- [x] 验证 Docker 已安装、Muliao 镜像已拉取、muliao.service 已 enabled
- [x] 配置 Tailscale：设备自动加入 tailnet，团队可远程 SSH
- [x] 运行 `deploy-team.sh muliao --start` 部署团队数据并启动服务
- [x] 验证 Hermes 通过 RPi 上的 Docker 容器响应 Telegram 消息

### Day 6：RPi 稳定性验证

**作为**团队，**我希望**RPi 5 上的 Hermes 能稳定运行且断电可恢复，**以便**作为可靠的部署目标。

**Tasks**：
- [ ] 运行 24 小时基础稳定性测试（内存、CPU、容器健康）
- [ ] 断电重启测试：拔电 → 重新通电 → 服务自动恢复 → Telegram 对话正常
- [ ] 记录 RPi 5 上的资源占用基线（内存、存储、CPU）
- [ ] 编写 `deploy/rpi/README.md`：团队内部 Phase A 部署操作手册

**验收标准**：
- ✅ `flash-ssd.sh` 一键完成镜像烧录 + cloud-init 注入
- ✅ RPi 5 首次开机后 cloud-init 自动完成所有初始化
- ✅ Hermes 在 RPi 5 上通过 Telegram 正常对话
- ✅ Tailscale 远程 SSH 可用
- ✅ 断电重启后所有服务自动恢复
- ✅ 24 小时运行无异常

**作为**用户（文哲），**我希望**发一条消息给 Hermes 就能搜索 arXiv 论文并获得中文摘要，**以便**节省每天手动刷论文的时间。

**Tasks**：

**验收标准**：

## Iteration 4：HR 对话式组建 v0（Day 13–15）

> **目标**：实现 Muliao 核心差异化——Hermes 通过多轮对话帮用户「招聘」新幕僚。这是最关键的 iteration。

### Day 13：HR 需求提取对话流程

**作为**用户，**我希望**告诉 Hermes「我需要一个帮我跟客户聊天的助手」后，Hermes 能通过几轮对话搞清楚我的需求，**以便**不用填写技术配置。

**Tasks**：
- [ ] 在 Hermes SOUL.md 中编写 HR 行为指引：
  - 识别「招聘」意图（「我想要一个…」「能不能帮我加一个…」「我需要…助手」）
  - 渐进式追问策略（初始 3–5 个核心问题：叫什么、做什么、什么风格、什么边界）
  - 不一次性问完，而是根据上下文动态追问
- [ ] 设计内部的需求提取模板（Hermes 用来组织对话结果的中间格式）
- [ ] 通过对话测试：模拟「招聘研究助手」「招聘客服助手」两个场景

### Day 14：需求→配置翻译 + 人话确认

**作为**用户，**我希望** Hermes 把对话结果翻译成配置后，用我能理解的话跟我确认，**以便**我知道新幕僚会怎么工作。

**Tasks**：
- [ ] 实现需求→配置映射逻辑（写入 Hermes SOUL.md / AGENTS.md 的 HR 指引）：
  - 用户描述 → SOUL.md（角色定义、行为风格、边界）
  - 用户提到的能力 → Skills 推荐（优先从 ClawHub 高人气 Skill 中匹配）
  - 用户提到的权限边界 → 初始权限配置
- [ ] 实现人话确认环节：Hermes 用非技术语言总结新幕僚的配置
  - 例：「小美会用亲切的语气跟客户聊，遇到报价会先问你，对吗？」
- [ ] 用户确认后，Hermes 输出完整的配置文件（SOUL.md + IDENTITY.md）

### Day 15：HR 流程端到端测试

**作为**用户，**我希望**从提出需求到确认配置的完整 HR 流程自然流畅，**以便**真正感受到「对话式组建」的价值。

**Tasks**：
- [ ] 端到端测试：模拟文哲场景——「帮我加一个每天帮我追踪论文的助手」
- [ ] 端到端测试：模拟丽姐场景——「我需要一个帮我接待客户的助手」
- [ ] 收集测试中发现的 prompt 问题，迭代 HR 指引

**验收标准**：
- ✅ 用户说「我需要一个…助手」→ Hermes 进入 HR 模式，开始渐进式追问
- ✅ 3–5 轮对话后，Hermes 输出人话版配置确认
- ✅ 用户确认后，Hermes 输出 SOUL.md + IDENTITY.md 内容
- ✅ 对话自然流畅，不像在填表

### 关键风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| ARM64 镜像兼容性问题 | Iteration 0–1 延期 | 提前在 QEMU 模拟验证，备选 x86 小主机方案 |
| RPi 5 NVMe 启动兼容性 | Iteration 1 受阻 | 确认固件版本，备选 SD 卡启动方案 |
| cloud-init 首次启动耗时过长 | Iteration 1 体验差 | 预拉取镜像写入 SSD，减少首次下载量 |

| DeepSeek R1 做 HR 对话质量不够 | Iteration 4 体验差 | 准备 Claude / GPT-4 作为 HR 角色的备选模型 |
| Gateway 重启导致服务中断 | Iteration 5 不稳定 | 实现配置校验 + 自动备份，必要时改为手动重启 |
| 研究助手深化依赖外部源稳定性 | Iteration 6 热点发现不准 | 优先用 arXiv 官方 API，HuggingFace/PwC 作为补充源 |

| WebSocket Relay 稳定性 | 设备断连 | Device Agent 指数退避重连；Telegram 为主通道， muliao.io 为辅助 |
