# 自然成长机制：HR 驱动演化详细设计

> 本文是 [design.md](../design.md) 八维差异 #3（自然成长机制）的展开文档。

Muliao 的自然成长不是让每个 Agent 自己记日记、自己复盘，而是由赫耳墨斯作为**独立的 HR 角色**，站在团队层面管理所有幕僚的成长。

---

## 与现有方案的本质区别

### 现有方案：单兵自我改进

ClawHub 上自我改进类 Skill 合计 503k 安装量，是社区第一刚需。它们的改进循环：

```
Agent 犯错 → Agent 自己记录到 .learnings/ → Agent 自己定期回顾 → Agent 自己晋升到 AGENTS.md
```

**改进的主体和客体是同一个 Agent**。没有外部视角，没有跨 Agent 学习，没有人类审核环节。

具体来说：

- **`self-improving-agent`（286k）**：被动记录——Agent 犯错后才学习。错误/纠正→结构化日志（`.learnings/`）→规则触发晋升（`Recurrence-Count >= 3` 且跨 2 个任务→自动写入 AGENTS.md）。
- **`proactive-agent`（115k）**：主动架构——WAL 协议（先写后回复）+ 三层记忆（SESSION-STATE / 日志 / 长期记忆）+ Heartbeat 自检 + 逆向提问。Agent 主动寻找改进机会，但仍是单兵行为。
- **`self-improving`（102k）**：自反思 + 自批评 + 自学习 + 记忆体系，模式与上述类似。

### Muliao 方案：团队级 HR 驱动演化

```
用户反馈 → 赫耳墨斯分析（组织级判断）→ 赫耳墨斯用人话提议变更 → 用户确认 → 配置生效
```

**改进的主体是赫耳墨斯（HR），客体是某个或多个幕僚，审批者是用户**。三个角色分离。

### 对照表

| | 单兵自我改进 | Muliao HR 驱动演化 |
|---|---|---|
| 改进主体 | Agent 自己 | 赫耳墨斯（独立 HR） |
| 观察视角 | 自我报告 | 外部观察 + 用户反馈 |
| 学习范围 | 单个 Agent 内部 | 跨 Agent 迁移 |
| 生效方式 | 自动（规则触发） | 审批制（用户确认） |
| 演化范围 | 行为调整 | 行为 + 团队结构 + 权限 |
| 新成员启动 | 从零开始 | 继承团队经验 |

## 五个核心机制

### 1. 外部观察式反馈采集

用户对小美说「这个报价单格式不对，应该先列总价再列明细」——这条反馈不是小美自己决定怎么处理，而是**赫耳墨斯截获并分析**：

- 这是小美个人问题？→ 更新小美的 SOUL.md（`报价单格式：总价在前、明细在后`）
- 这是团队共性问题？→ 所有涉及文档输出的幕僚都需要更新
- 这说明小美缺少某个 Skill？→ 赫耳墨斯推荐安装合适的 Skill

单兵模式下，小美只会记录「报价单格式错了」；HR 模式下，赫耳墨斯会做出**组织级判断**。

#### 反馈来源

| 来源 | 示例 | 赫耳墨斯行为 |
|------|------|-------------|
| 用户显式纠正 | 「不对，应该这样……」 | 分析是个人问题还是共性问题，提议配置变更 |
| 用户隐式偏好 | 用户反复手动修改某个格式 | 识别模式，主动提议写入幕僚 SOUL.md |
| 幕僚执行失败 | 工具调用报错、任务超时 | 诊断原因，决定是调参、换 Skill 还是调整分工 |
| 幕僚间协作摩擦 | 交接信息丢失、重复工作 | 调整协作协议或职责边界 |

### 2. 跨 Agent 学习迁移

小李（研究员）在搜集竞品信息时发现了一个高效的搜索策略。在单兵模式下，这个经验只存在于小李的 `.learnings/` 里。

HR 驱动模式下：

1. 赫耳墨斯识别到这个经验对小王（市场分析师）也有价值
2. 用人话向用户确认：「小李找到了一个很好的竞品分析方法，小王做市场报告时也能用上，要不要让小王也学会？」
3. 用户同意后，赫耳墨斯将相关经验翻译成小王的 SOUL.md 更新

**单 Agent 系统做不到这一点——每个 Agent 是孤岛，经验无法流动。**

#### 迁移决策流程

```
幕僚 A 产生有效经验
        ↓
赫耳墨斯评估：这条经验是否适用于其他幕僚？
        ↓
    ┌── 仅适用于 A → 写入 A 的 SOUL.md
    ├── 适用于特定幕僚 → 人话确认后分别写入
    └── 团队通用 → 写入团队级共享配置
```

### 3. 审批制生效

`self-improving-agent` 的晋升规则是：`Recurrence-Count >= 3` 且跨 2 个任务 → **自动晋升**到 AGENTS.md。没有人类参与。

Muliao 的流程：

```
用户反馈 → 赫耳墨斯分析 → 赫耳墨斯用人话提议变更 → 用户确认 → 配置生效
```

例如赫耳墨斯会说：

> 「小美最近处理退货时总是忘记确认订单号，我建议在她的工作流程里加上『先确认订单号再处理退货』这一步，你觉得呢？」

用户说「好」，赫耳墨斯才更新小美的 SOUL.md。用户说「不用，那个场景特殊」，变更就不会发生。

这保证了用户的**掌控感**（设计理念 #4），同时降低了自动演化可能带来的风险——特别是在涉及客户沟通、资金操作等敏感场景时。

#### 审批分级

| 变更类型 | 风险等级 | 审批方式 |
|----------|---------|---------|
| 措辞/格式偏好 | 低 | 赫耳墨斯简要告知，默认生效，用户可撤回 |
| 工作流程调整 | 中 | 赫耳墨斯提议，等待用户明确确认 |
| 新增 Skill / 工具权限 | 高 | 赫耳墨斯详细说明影响，必须用户批准 |
| 团队结构变更（新增/合并角色） | 高 | 赫耳墨斯给出方案 + 理由，用户批准后执行 |

### 4. 团队结构演化

单兵自我改进只能改变**一个 Agent 的行为**。HR 驱动演化还能改变**团队组成**：

- **角色新增**：「你最近让小美做了很多数据分析的活，但她本来是客服。要不要我招一个专门做数据分析的幕僚？」
- **角色合并**：「小李和小王的工作越来越重叠了，他们可以合并成一个角色，你觉得呢？」
- **权限调整**：「小美这周处理了 50 个客户对话都没出问题，要不要让她自己决定何时发资料？」（联动渐进式信任 #5）
- **职责重新分配**：「小美现在接的客户太多了，响应变慢。要不要把 VIP 客户分给一个新幕僚专门跟进？」

这是**组织级演化**——超出了任何单 Agent 改进方案的范畴。

#### 触发条件

| 信号 | 赫耳墨斯判断 | 可能的结构变更 |
|------|-------------|--------------|
| 某幕僚频繁处理非本职任务 | 角色边界模糊 | 建议新增专职幕僚 |
| 两个幕僚产出高度重叠 | 职责冗余 | 建议合并角色 |
| 某幕僚任务排队过长 | 负载不均 | 建议拆分角色或分流 |
| 用户反复将同类任务手动重定向 | 路由规则不匹配 | 建议调整分工或绑定规则 |
| 幕僚持续零错误运行 | 信任积累 | 建议放权（联动 #5） |

### 5. 新成员入职加速

当赫耳墨斯「招聘」新幕僚时，已有团队成员的经验可以作为启动包：

- 小美做客服积累了「客户最常问的 10 个问题」→ 新招的客服幕僚直接继承
- 团队级的共性教训（「发文件前一定要确认格式」）→ 写入所有新幕僚的初始 SOUL.md
- 特定 Skill 组合已在类似角色验证过 → 新幕僚直接安装

单兵系统的新 Agent 从零开始；HR 系统的新 Agent **站在团队经验上起步**。

#### 入职启动包组成

```
新幕僚初始配置
├── SOUL.md
│   ├── 角色定义（从 HR 对话中提取）
│   ├── 团队共性规范（从团队级配置继承）
│   └── 同类角色积累的经验（从已有幕僚迁移）
├── skills/
│   └── 赫耳墨斯推荐的 Skill 组合（基于角色 + ClawHub 热门匹配）
└── memory/
    └── 相关领域的已有知识摘要（从团队知识库提取）
```

## 实现路径

### v0（MVP 阶段）

最小可用版本，验证核心循环：

1. **反馈采集**：赫耳墨斯监听用户对幕僚的显式纠正（「不对」「应该是」「改成」等触发词）
2. **变更提议**：赫耳墨斯用人话总结需要改什么，问用户是否同意
3. **配置写入**：用户确认后，赫耳墨斯修改目标幕僚的 SOUL.md
4. **生效确认**：赫耳墨斯告知用户「已更新，下次小美会按新方式处理」

v0 不做跨 Agent 迁移、不做团队结构演化、不做自动化审批分级。

### v1

在 v0 基础上增加：

- 跨 Agent 学习迁移（机制 #2）
- 审批分级（低风险默认生效，高风险等待确认）
- 新成员入职启动包（机制 #5）

### v2

完整版：

- 团队结构演化建议（机制 #4）
- 隐式偏好识别（用户没有明说但行为暗示的偏好）
- 与渐进式信任（#5）联动
- 演化历史可视化（哪个幕僚因为什么原因做了哪些调整）

## 与 `self-improving-agent` / `proactive-agent` 的关系

Muliao 的 HR 驱动演化和这些单兵 Skill **不是替代关系，而是互补关系**：

- 每个幕僚内部仍然可以安装 `self-improving-agent` 来记录自己的错误日志——这是**个人笔记本**
- 赫耳墨斯的 HR 演化是在此之上的**组织级管理**——它审查多个幕僚的表现，做出跨团队的决策

类比公司：员工自己记工作日志（`self-improving-agent`），不影响 HR 做绩效评估和组织调整（赫耳墨斯）。两者层级不同，可以共存。

## 实现难点与风险

### 各机制实现难度评估

| 机制 | 难度 | 说明 |
|------|------|------|
| 1. 外部观察反馈采集 | ★★☆☆☆ | 赫耳墨斯是主 Agent，所有消息经过它，检测靠 prompt 层面实现 |
| 2. 跨 Agent 学习迁移 | ★★☆☆☆ | 文件系统读写，纯文件操作 |
| 3. 审批制激活 | ★☆☆☆☆ | 正常对话流程，不需要新基础设施 |
| 4. 团队结构演化 | ★★★☆☆ | 需修改 `openclaw.json` + 创建 workspace 目录 + 网关重启 |
| 5. 新成员入职加速 | ★★☆☆☆ | 读取已有文件 → LLM 蒸馏 → 写入新文件 |

### 关键挑战

#### 1. 反馈拦截时机（★★★）

如果用户通过 channel 直接与子 Agent 对话（OpenClaw 支持将 channel 绑定到特定 Agent），赫耳墨斯**无法实时看到对话内容**。

两种方案：

- **方案 A：所有对话经由赫耳墨斯路由**——赫耳墨斯接收所有消息并转发给对应幕僚，代价是多一跳延迟
- **方案 B：事后审阅**——赫耳墨斯通过 Heartbeat 定时任务 + `sessions_history` API 回溯各幕僚的对话记录

v0 推荐方案 A：当前只有单 Agent（赫耳墨斯），所有对话天然经过它，不存在拦截问题。等 v1 引入子 Agent 后再解决。

#### 2. LLM 判断质量（★★★★）

这是**最大的不确定性**。HR 演化的核心决策完全依赖 LLM 的判断力：

- 用户纠正是个人偏好还是团队共识？（应该迁移到其他 Agent 吗？）
- 某个经验值不值得提炼为规则？（避免过度演化，频繁修改 SOUL.md 反而让 Agent 行为不稳定）
- 新成员入职启动包应该包含什么？（过多 → 角色模糊，过少 → 没用）

这不是工程问题而是**Prompt Engineering + 模型能力**问题。当前使用的 DeepSeek R1 在这类细粒度判断上可能偏弱，Claude / GPT-4 级别模型会更适合 HR 角色。

#### 3. 网关重启的破坏性（★★★）

机制 #4（团队结构演化）需要修改 `openclaw.json` 并重启 OpenClaw 网关：

- 进行中的会话可能被中断
- 配置错误可能导致整个系统不可用
- 回滚不方便（需要保留配置快照）

建议 v0 阶段**完全跳过机制 #4**，v1 引入时增加配置校验和自动备份。

#### 4. 上下文窗口压力（★★★）

赫耳墨斯启动时已经加载 `SOUL.md` + `USER.md` + `MEMORY.md` + 当日日志，加上 HR 监控指令会进一步膨胀 prompt。

- DeepSeek R1 的 200K 上下文窗口在容量上足够
- 但 prompt 膨胀会推高 token 成本，且 LLM 在长 prompt 中的判断质量会下降（lost in the middle 问题）
- 需要考虑 HR 指令的精简和分层——不是所有对话都需要全量 HR 监控

### v0 最小验证步骤

只需 3 步即可验证核心假设：

1. **在 `SOUL.md` 中追加 HR 行为指引**——告诉赫耳墨斯"当用户纠正你时，记录到 memory 并提议修改自己的配置"
2. **在 `AGENTS.md` 中追加反馈闭环规则**——定义触发词、记录格式和变更审批流程
3. **实际对话测试**——故意在对话中纠正赫耳墨斯，观察它是否能识别反馈、提议变更并在确认后修改自身配置

这 3 步不需要新代码、不需要改 Docker 镜像、不需要装额外 Skill，只需要编辑两个 Markdown 文件。
